## Project Overview
Enterprise Support Knowledge Assistant: an agentic RAG system over LangChain/LangGraph documentation. Built as a portfolio project for AI/ML Engineer and Generative AI Engineer job applications — the goal is to demonstrate production RAG skills (hybrid retrieval, agentic routing, MCP tool integration, formal evaluation), not just a basic "chat with docs" demo.

## Tech Stack
- Python 3.11+
- LangGraph for agent orchestration, LangChain for retrieval utilities
- ChromaDB (local, no cloud account) as the vector store
- `BAAI/bge-small-en-v1.5` for embeddings (384-dim, local, free — swapped from all-MiniLM-L6-v2 because its 256-token hard limit silently truncates code-block-heavy chunks; bge-small's 512-token window fits a code example + explanation together)
- `rank-bm25` for sparse retrieval, `cross-encoder/ms-marco-MiniLM-L-6-v2` for reranking
- **LLM: `llama3.1:8b` served locally by [Ollama](https://ollama.com)** — swapped from Gemini after hitting the free-tier 20-RPM cap during Phase 6 RAGAS eval. Local means no rate limits, no 503s, no API key, and it fully honors the "free-tier, locally runnable" project ethos. Runs on ~5 GB VRAM (RTX 4050-class card). Same model serves as RAG generator AND RAGAS judge.
- MCP Python SDK for the tool server
- RAGAS for evaluation, plus a custom routing/tool-correctness metric (Jaccard-based) to score whether the agent chose the right path (retrieve / get_package_version / get_corpus_status / fetch_live_doc / clarify — see `eval/routing_testset.json`)
- LangSmith for tracing and observability (native LangChain/LangGraph integration, free tier)
- LangGraph's built-in `MemorySaver` checkpointer for multi-turn conversation memory
- FastAPI backend; frontend TBD (React or Streamlit for v1)
- `python-frontmatter` for stripping and extracting YAML frontmatter from MDX/MD files

## Key Decisions — don't relitigate these without discussion
- **Corpus:** `langchain-ai/docs` (MIT licensed), pinned to commit `22cbff9d7ad4b676db836360d98adc343f523ee1` (2026-07-03) — this is the ACTUAL commit checked out in `data/raw/` and reported by the manifest/`get_corpus_status`; an earlier `662d399…` claim was wrong (that SHA isn't even in `data/raw` history). Scope is `src/oss/langchain/` + `src/oss/langgraph/` + `src/oss/concepts/` only (≈120 in-scope files → ~116 kept after the stub filter) — explicitly excludes `src/oss/python/integrations/` (579 files, per-partner integration catalog, not differentiated conceptual content) and `src/oss/javascript/` (179 files, duplicate content in a second language). Verified by cloning directly: no `.ipynb` notebooks exist anywhere in this repo — it's a pure Mintlify `.mdx` site.
- **MCP tools (Phase 7) return REAL data, not synthetic.** Superseded the earlier `get_ticket_status`/`get_product_version` synthetic-ticket idea (too generic, not tied to a docs assistant). There are **THREE** tools: (1) `get_package_version(package)` — the CURRENT version of a package from the **live PyPI** API (free, no key), motivated by the docs being frozen at a pinned commit so retrieval can't answer version questions; (2) `get_corpus_status(topic?)` — the docs' pinned commit / freshness / per-product & per-doc_type coverage read from our **own manifest** (fully local); (3) `fetch_live_doc(query?, path?)` — fetches a CURRENT LangChain/LangGraph doc page live from GitHub for topics the FROZEN corpus can't answer (staleness fix): query mode ranks in-scope live `.mdx` paths with bge-small and gates at `min_score=0.62` (returns `matched:false` + candidates when out of scope so the agent refuses honestly), path mode fetches an exact file. It inlines snippet code over GitHub (same code-preservation as the local cleaner). No `mcp_server/mock_data/` folder. Server is `mcp_server/server.py` (FastMCP, stdio; prewarms bge-small + loads it HF-offline so a fresh subprocess doesn't stall); logic in `mcp_server/tools.py`; the agent connects via `agent/mcp_client.py` (`warm_tools()`). Routing contrast for Phase 8: docs question → retrieve; "latest version?" → get_package_version; "how fresh / do you cover X?" → get_corpus_status; new/out-of-scope doc content → fetch_live_doc.
- **Eval set** is generated from our own corpus chunks (RAGAS testset generation), not an external QA dataset — keeps faithfulness/relevance scores meaningful against our actual retrieval.
- **Build philosophy:** get a naive, linear RAG pipeline fully working end-to-end first (ingest → embed → dense retrieve → generate) *before* adding hybrid retrieval, MCP, or LangGraph agent routing. One layer at a time, validated before the next is added — do not parallelize all components.
- **Data cleaning is a dedicated phase (Phase 1) that runs before ingestion.** `data/raw/` is never modified — it is read-only source-of-truth. All cleaned, normalized output lands in `data/cleaned/` as plain `.md` files alongside a `manifest.json` that tracks format, SHA256 hash, and processing status per file. Ingestion (Phase 2) reads from `data/cleaned/`, not `data/raw/`.
- **Relevance filtering happens at clean time, not chunk time.** Only `src/oss/langchain/`, `src/oss/langgraph/`, and `src/oss/concepts/` are passed through to `data/cleaned/`. LangSmith docs, the `python/integrations/` catalog, the `javascript/` tree, build tooling, fonts, images, config files, and navigation-only pages are all excluded at the detector step.

## Conventions
- Every chunk's metadata must include `source_file`, `section`, `chunk_id` — kept generic on purpose so additional sources (e.g. PDFs) can be added later without a schema migration.
- Test and validate each phase in isolation before wiring it into the next layer (e.g. test the MCP server directly before wiring it into the LangGraph agent).
- No cloud vector DB signups, no paid API dependencies beyond the LLM itself — this stays a free-tier, locally runnable project.
- The eval corpus includes a deliberate "noisy data" subset (10–15 irrelevant chunks mixed into the real corpus) to stress-test that retrieval and guardrails correctly ignore off-topic content instead of hallucinating from it.
- **`data/raw/` is read-only.** Never write to or modify it. The cleaning pipeline reads from it and writes normalized `.md` files to `data/cleaned/`.
- **`data/cleaned/manifest.json`** tracks every processed file: `source_rel_path`, `target_rel_path`, `original_format`, `content_hash` (SHA256 of the RAW source), `source_commit`, `ingested_at`, `status` (ok / dropped:<reason> / error), plus derived `title`, `product`, `doc_type`, `section`. On re-run, only files whose hash changed are reprocessed — this is what makes incremental upserts (Phase 3) work without a full reindex. CAVEAT: `content_hash` is of the RAW source only, so when the **pipeline** changes (normalizer/chunker), the hash doesn't move and a plain re-run won't detect it — rebuild with `--force` (cleaner) / re-chunk / embed `--force`.
- **`doc_type` is derived from the docs' OWN navigation** (`data/raw/src/docs.json`), not guessed from the path (the repo organizes by topic, not doc-type). `ingestion/cleaner/filter.py` maps each page's nav group → a canonical type (concept / how-to / reference / tutorial / get-started / integration / error). This is what makes `get_corpus_status`'s coverage breakdown meaningful (0 "unknown").
- The cleaner uses a **format-registry pattern** (`{".mdx": clean_mdx, ".md": clean_markdown}`) so new source formats (e.g. `.ipynb`, `.rst`, `.pdf`) can be added later by registering one function, not by touching the pipeline — the current corpus has zero `.ipynb` files, so no notebook handler is needed yet.
- Cleaned files must be verifiable independently of the ingestion pipeline — print 5 random files from `data/cleaned/` and they should be plain prose Markdown with no frontmatter, no JSX, no nav/sidebar cruft, and no "edit this page" links.
- **Never lose code examples.** Many pages keep their code in imported Mintlify snippet components (`import X from '/snippets/...'` rendered as `<X />`, often inside `:::python` blocks). The normalizer (`ingestion/cleaner/normalizer.py`) **inlines these snippets first** (via an injected resolver) so the code survives cleaning — stripping the `<X />` tag without inlining would silently delete exactly the code a code-focused retriever needs. `fetch_live_doc` resolves snippets over GitHub for the same reason. `:::js`/`:::ts` variants are still dropped (Python-native project).
- **Token counting anywhere in the pipeline (chunking, budget checks) must use the actual embedding model's own tokenizer** (`AutoTokenizer.from_pretrained("BAAI/bge-small-en-v1.5")`), never a different model's tokenizer (e.g. tiktoken/cl100k_base) as an approximation — different tokenizers segment text differently enough that "400 tokens" in one is not "400 tokens" in another.
